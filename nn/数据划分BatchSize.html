第三步：优化神器 —— 指数加权平均 (EMA / Momentum)

为了防止模型“一惊一乍”，我们引入了类似“动量”的概念。核心思想是：不要只听今天这批数据的，要结合过去几天的平均经验。

它的数学公式是这样的（我们假设动量系数 $\beta = 0.5$）：

$$V_t = \beta \cdot V_{t-1} + (1 - \beta) \cdot \theta_t$$
  • $V_t$ ：模型今天要走的最终实际步伐。
  • $\beta$ ：对过去经验的信任度。如果是 $0.5$，代表 50% 听过去的。
  • $V_{t-1}$ ：昨天积攒下来的经验步伐。
  • $(1 - \beta)$ ：对今天新数据的信任度。如果是 $0.5$，代表 50% 听今天的。
  • $\theta_t$ ：今天这批数据计算出来的新梯度（就是上表的数字）。

第四步：Batch Size 究竟该怎么选？（大与小的博弈）

在实际炼丹中，Batch Size 的大小直接决定了梯度下降的方向准确度和训练效率。这是一个在“稳定”与“泛化”之间寻找平衡的过程。

Batch Size 设定 优势 (Pros) 劣势 (Cons)
较大 (Large) 梯度计算更准确且稳定，训练曲线平滑。
能充分利用 GPU 矩阵运算的并行加速能力，单次 Epoch 的计算耗时极短。		 泛化能力可能稍弱，容易陷入尖锐的局部最小值 (Sharp Local Minima)。
对显存要求极高，容易导致 OOM (Out of Memory)。
较小 (Small) 梯度包含较大噪声，这种随机性有助于模型跳出局部最优解。
通常在测试集上能获得更好的泛化性。		 训练过程极不稳定，损失函数震荡剧烈。
无法有效发挥 GPU 的并行矩阵运算优势，整体收敛速度慢。
💡 炼丹经验值： 业界常用的 Batch Size 经验值为 32, 64, 128。如果你的显卡显存不够，可以尝试降到 16 或 8。
注意：通常建议将数值设置为 2 的 N 次方，这与 GPU 的底层硬件架构（如 CUDA 线程束大小）和内存对齐有关，能最大化硬件计算效率。

第五步：🔥 核心面试题自测

理解了底层的迭代原理和博弈过程，面对大厂的这 5 道高频基础面试题自然水到渠成：

  1. Epoch、Batch Size 和 Iteration 的区别与联系是什么？ 解析： Epoch 是指完整遍历一次整个数据集；Batch Size 是单次执行前向/反向传播时所使用的样本数量；Iteration 是指进行一次参数更新（即跑完一个 Batch）。三者的关系公式为：$Iteration = \frac{\text{Dataset Size}}{\text{Batch Size}}$。
  2. 为什么 Batch Size 通常设定为 2 的次幂（如 32, 64, 128）？ 解析： 主要是为了匹配 GPU 等底层硬件设备的架构（例如 CUDA 的线程束 Warp 大小通常是 32）。符合 2 的次幂能够实现更优的内存对齐，大幅提高矩阵乘法的并行计算效率。
  3. 如果训练时 Loss 曲线震荡非常剧烈，可能的原因是什么？应该如何调整？ 解析： 核心原因通常是 Batch Size 设置过小（梯度噪声过大导致盲目乱走），或者学习率（Learning Rate）设置过高。调整策略：适当增大 Batch Size、降低学习率，或者在优化器中引入动量（Momentum）机制进行平滑。
  4. 在极小的 Batch Size 下，为什么模型的泛化能力往往反而更好？ 解析： 小 Batch 带来的剧烈梯度波动包含大量随机噪声，这种噪声起到了一种隐式的正则化（Regularization）作用，使得模型在优化空间中不断“跳跃”，从而不容易落入训练集的尖锐局部极小值（Sharp Minima），更倾向于停留在平缓的极小值区域，因此在未知测试集上表现更佳。
  5. 如果服务器显存极小，但又想使用大 Batch Size 来稳定训练，有什么工程手段？ 解析： 可以使用梯度累加（Gradient Accumulation）技术。即保持显存能承受的极小 Batch（如 4）进行前向和反向传播，但不立即更新参数，而是将这几次计算出的梯度在内存中累加。当累积了足够的步数（比如累加 8 次等价于 Batch Size 32）后，再统一执行一次优化器的 `step()` 更新参数并清零梯度。这样本质是用时间换取空间。
